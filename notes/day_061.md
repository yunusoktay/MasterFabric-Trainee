## Outcome

Day 61 introduces the BLoC (Business Logic Component) pattern in Flutter.

### What is BLoC?

BLoC is a state management pattern that separates:

- **UI layer**: widgets and rendering
- **Business logic layer**: decision-making and state transitions

Instead of calling `setState()` directly in UI widgets, the UI sends an **Event** to a BLoC, and the BLoC emits a new **State**.

Flow:

`User action -> Event -> BLoC -> New State -> UI rebuild`

### Core Concepts

#### Event

An Event represents something that happened (usually user action).

Examples in the counter app:

- `CounterIncrementPressed`
- `CounterDecrementPressed`

#### State

A State represents the UI data at one point in time.

In the counter app:

- `CounterState(value)`

#### BLoC

The BLoC receives events, runs business logic, and emits next states.

In the counter app:

```dart
class CounterBloc extends Bloc<CounterEvent, CounterState> {
  CounterBloc() : super(const CounterState(0)) {
    on<CounterIncrementPressed>((event, emit) {
      emit(CounterState(state.value + 1));
    });

    on<CounterDecrementPressed>((event, emit) {
      emit(CounterState(state.value - 1));
    });
  }
}
```

### Why Streams Matter

BLoC works on top of streams:

- Events are fed into the BLoC
- States are emitted over time as a stream
- Widgets listen and rebuild when new states arrive

This is why Day 11-15 stream knowledge is important.

### flutter_bloc in Practice

The app uses:

- `BlocProvider` to provide `CounterBloc` to widget tree
- `context.read<CounterBloc>().add(...)` to send events
- `BlocBuilder<CounterBloc, CounterState>` to rebuild UI from state

## Key Concepts

| Concept | What It Means |
|---|---|
| **BLoC** | Component that maps Events to States |
| **Event** | Input action sent to BLoC |
| **State** | Output data emitted by BLoC for UI |
| **BlocProvider** | Injects BLoC into widget tree |
| **BlocBuilder** | Rebuilds UI based on current state |
| **flutter_bloc** | Package with widgets/tools for BLoC pattern |

## setState vs BLoC

| `setState` approach | BLoC approach |
|---|---|
| Logic mixed inside widget | Logic moved into BLoC |
| Harder to test in isolation | Easy to unit test BLoC |
| Local/simple state only | Scales better for complex apps |
| Direct UI mutation | Event-driven, predictable transitions |

## Common Mistakes to Avoid

- Putting business logic back into widgets instead of BLoC
- Emitting mutable states (prefer immutable state classes)
- Triggering side effects directly inside `build()` methods
- Creating multiple BLoC instances unintentionally
- Forgetting to model actions as explicit events

## Why It Matters

As apps grow, separating UI and business logic becomes essential. BLoC gives a predictable, testable, and scalable architecture. Day 61 is the foundation before moving to more advanced topics like async API calls in BLoC, loading/error states, and repository integration.
