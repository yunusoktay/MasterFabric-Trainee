## Run

## Outcome

Day 13 introduces the `async` / `await` syntax, which is syntactic sugar for the `.then()` / `.catchError()` chains from Day 12.

### async / await Basics

- A function marked with `async` **always returns a `Future`**, even if you write `return 42;` the caller receives a `Future<int>`.
- `await` can only be used inside an `async` function.
- `await someFuture` **pauses** the function until that Future completes, then unwraps the value into a regular variable.
- The thread is never blocked — only the function pauses, the event loop keeps running, so the UI stays responsive.

```dart
Future<String> fetchUsername() async {
  await Future.delayed(const Duration(milliseconds: 200));
  return 'yunus';
}

Future<void> main() async {
  final String name = await fetchUsername();
  print(name);
}
```

### .then() vs. async/await

Same logic, two different styles:

```dart
fetchUsername().then((name) {
  return fetchAge().then((years) {
    print('$name is $years years old');
  });
});
```

```dart
final String name = await fetchUsername();
final int years = await fetchAge();
print('$name is $years years old');
```

The `async` / `await` version reads top-to-bottom, with no nested callbacks. Deeply nested `.then()` chains are often called "callback hell"; `await` removes that problem.

### Error Handling

Instead of `.then().catchError().whenComplete()`, use the familiar sync-style `try / catch / finally`:

```dart
try {
  final int value = await riskyOperation();
  print('success: $value');
} on StateError catch (error) {
  print('StateError: ${error.message}');
} catch (error) {
  print('generic: $error');
} finally {
  print('always runs');
}
```

- An exception thrown inside a Future is re-thrown at the `await`.
- `on XType catch (e)` catches a specific exception type.
- The `finally` block always runs, regardless of success or error.

### Sequential vs. Parallel

Awaiting one by one makes the calls run **in sequence**:

```dart
final a = await slowTask('A', 200); // 200 ms
final b = await slowTask('B', 200); // +200 ms
final c = await slowTask('C', 200); // +200 ms
// total ~600 ms
```

When the calls are independent, start them in parallel with `Future.wait`:

```dart
final results = await Future.wait<String>([
  slowTask('A', 200),
  slowTask('B', 200),
  slowTask('C', 200),
]);
// total ~200 ms
```

The Day 13 demo shows the difference clearly: **~600 ms** sequential vs. **~200 ms** parallel.

## Key Concepts

| Keyword / API | What It Does |
|---|---|
| **`async`** | Marks a function as asynchronous; its return type becomes `Future<T>` |
| **`await`** | Pauses the `async` function until the Future completes |
| **`try / catch`** | Catches an error thrown during an `await` |
| **`on XType catch`** | Catches a specific exception type |
| **`finally`** | Always runs, whether the Future succeeded or failed |
| **`Future.wait([...])`** | Awaits multiple Futures in parallel |

## Then/Await Equivalents

| `.then()` style | `async/await` equivalent |
|---|---|
| `f().then((v) => ...)` | `final v = await f();` |
| `.catchError((e) => ...)` | `try { ... } catch (e) { ... }` |
| `.whenComplete(() => ...)` | `finally { ... }` |
| `Future.wait([...])` | Same API: `await Future.wait(...)` |

## Why It Matters

In Flutter, the UI runs on a single thread. Running a long operation (network request, file read) synchronously would freeze the screen. `async` / `await` lets those tasks run in the background while keeping the code readable like synchronous code — the UI stays smooth and the code stays maintainable.
